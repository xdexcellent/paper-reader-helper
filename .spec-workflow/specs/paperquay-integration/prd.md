# PRD: PaperQuay 渐进集成到 paper-reader-helper

## 1. 背景

当前项目是 FastAPI + React/Vite 的 Web 应用，已有论文导入、上传、MinerU 解析、摘要、embedding、订阅、每日速览、推荐和 AI 助手能力。用户希望把当前“论文管理”替换为 PaperQuay 风格的文献管理体验。

PaperQuay 官方定位是“桌面端优先的 AI 辅助文献管理软件，面向 PDF 阅读、全文翻译、论文概览和 Agent 工作流”。其 README 描述的核心能力包括：本地 SQLite 文献库、PDF 导入确认、分类/标签/元数据编辑、PDF 阅读、MinerU 结构块、全文/块级翻译缓存、结构化论文概览、Agent 文库操作和 Zotero 导入。

参考来源：
- GitHub: https://github.com/WangQrkkk/PaperQuay
- 中文 README: https://github.com/WangQrkkk/PaperQuay/blob/main/README.zh-CN.md
- 许可: AGPL-3.0-only

## 2. 产品决策

选择方案 2：逐步把 PaperQuay 的文献管理体验和关键能力移植到当前项目，保留现有 FastAPI 后端和当前 Web 应用能力。

本 PRD 不默认直接复制 PaperQuay 源码。由于 PaperQuay 使用 AGPL-3.0-only，任何源码级复用都必须先完成许可决策；在许可未确认前，只参考公开产品能力、信息架构和交互模式。

## 3. 目标

- 用 PaperQuay 式“文库优先”界面替换当前“论文管理”主体验。
- 保留当前项目已有的订阅、每日速览、推荐、解析、摘要、embedding 和 AI 助手能力。
- 增强文献库能力：导入确认、元数据管理、分类树、标签、收藏、阅读状态、搜索筛选。
- 为后续 Reader、MinerU block 联动、翻译缓存、Agent 文库操作和 Zotero 导入预留清晰边界。
- 在编码前通过 200 道需求填写题明确范围，避免一次性大重构失控。

## 4. 非目标

- 第一阶段不把整个项目改成 Tauri/Rust 桌面应用。
- 第一阶段不直接替换数据库为 PaperQuay schema，除非问卷确认允许迁移。
- 第一阶段不默认复制 PaperQuay 源码。
- 第一阶段不做云同步、多人协作、复杂权限体系。
- 第一阶段不承诺完整 PDF 高级批注和导出，除非问卷确认提升优先级。

## 5. 用户画像

- 个人研究者：需要快速筛选、分类、阅读和总结论文。
- 学生/论文写作者：需要保存论文概览、翻译重点段落、保留笔记。
- AI 辅助研究用户：希望让 Agent 帮助整理标签、补元数据、批量分类和总结。

## 6. 核心使用旅程

1. 用户进入“论文管理”，看到 PaperQuay 风格文库工作区。
2. 用户导入 PDF，先进入元数据确认页。
3. 系统复制/移动/保留 PDF，并创建文献记录。
4. 用户通过分类树、标签、搜索和筛选管理论文。
5. 用户打开论文，查看元数据、概览、PDF/Markdown 阅读内容。
6. 用户触发 MinerU 解析、摘要、embedding，任务状态可追踪。
7. 后续阶段中，用户可查看结构化 blocks、缓存译文、批注、Agent 操作和 Zotero 导入结果。

## 7. 分阶段范围

### Phase 0: 需求冻结

- 完成 `requirements-questionnaire.md` 中 P0/P1 问题。
- 决定许可策略：仅参考体验，或允许源码级复用并接受 AGPL 义务。
- 决定第一阶段交付形态：Web-only，还是为桌面化预留接口。

### Phase 1: 文库工作区替换

- 新建 PaperQuay 风格文库页面，替换当前“论文管理”默认入口。
- 保留当前 API：`/papers`、`/papers/upload`、`/papers/{id}`、`/papers/{id}/parse`、`/papers/{id}/summarize`、`/papers/{id}/embed`。
- 增加导入确认体验，但尽量复用现有上传和存储服务。
- 文库三栏：分类树、论文列表、详情/操作区。
- 增加元数据编辑、收藏/阅读状态/笔记等字段的设计预留；是否落库由问卷确认。

### Phase 2: 阅读器和论文概览

- 详情区升级为 PaperQuay 式概览面板。
- 将当前 summary 字段映射到背景、问题、方法、实验、发现、结论、局限等概览字段。
- 阅读器支持 PDF/Markdown 切换。
- 保持解析失败时 PDF 仍可阅读。

### Phase 3: MinerU block 与翻译

- 保存 MinerU 结构化 block。
- 支持 block 与 PDF 页区域联动。
- 支持划词翻译、block 级翻译缓存和译文跳转。

### Phase 4: Agent 和 Zotero

- AI 助手升级为文库 Agent 工作区。
- Agent 操作必须具备人工确认、执行轨迹、失败回滚或审计。
- 支持 Zotero 只读导入，复制 `zotero.sqlite` 到临时文件读取，不修改原库。

## 8. 功能需求摘要

- 文库：分类树、论文列表、详情、搜索、筛选、标签、收藏、阅读状态。
- 导入：PDF 拖拽/选择、导入确认、重复检测、元数据补全预留。
- 文件：存储根目录、复制/移动/保留路径策略、文件缺失提示。
- 解析：复用当前 MinerU 任务，预留 block 存储。
- 概览：结构化字段，支持生成、重试、编辑预留。
- 阅读：PDF 内嵌查看、Markdown 阅读、章节导航预留。
- 翻译：OpenAI-compatible endpoint，缓存策略预留。
- Agent：文库操作工具、确认机制、轨迹展示。
- Zotero：只读导入、collection/tag 映射、错误报告。

## 9. 非功能需求

- 性能：第一阶段在 1k 篇论文内列表和筛选保持可用；更大规模再引入虚拟列表/索引优化。
- 安全：继续保护所有 API；不在前端暴露 API key；删除和批量操作需要确认。
- 隐私：PDF、数据库、解析结果、API key 不提交到仓库。
- 可靠性：解析/摘要/embedding 失败不阻塞阅读；任务状态可恢复。
- 可访问性：表单有 label，按钮有明确名称，键盘可达，焦点状态可见。
- 视觉：采用数据密集型文库工作台，不做营销页；不使用单一紫色主题；遵循当前应用的深色/浅色体系。

## 10. 成功指标

- 旧“论文管理”核心路径可在新文库界面完成：导入、选择论文、查看详情、解析、摘要、embedding、删除。
- 每日速览和推荐页打开论文时能进入新文库详情或阅读器。
- 现有后端论文相关测试仍通过。
- 新增前端测试覆盖文库三栏、导入确认、详情操作、失败状态。
- 数据迁移前后论文数量和关键字段一致。

## 11. 主要风险

- 许可风险：PaperQuay 为 AGPL-3.0-only，源码复用可能触发开源义务。
- 架构差异：PaperQuay 是 Tauri + Rust + SQLite，当前项目是 Web + FastAPI + SQLModel。
- 范围膨胀：阅读器、翻译、Agent、Zotero 都是大功能，必须分阶段。
- 数据迁移风险：现有 schema 较简单，直接迁移到完整文献库模型可能影响历史数据。
- UI 复杂度：文库工作台信息密度高，需要控制组件边界和测试覆盖。

## 12. 编码前门禁

编码前必须满足：

1. `requirements-questionnaire.md` 至少完成所有 P0/P1 问题。
2. `requirements.md` 经用户确认。
3. 技术设计经用户确认。
4. 拆分后的任务清单包含测试和回滚策略。
5. 许可策略明确写入任务范围。
