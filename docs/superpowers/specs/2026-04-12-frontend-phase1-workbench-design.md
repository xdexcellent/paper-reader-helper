# 前端 Phase 1 改版设计稿：浅色可操作工作台

## 1. 目标

在不推翻当前 MVP 前后端闭环的前提下，把现有“偏静态的深色阅读页”升级为一个**浅色、可操作、可反馈**的论文处理工作台。

本次改版只覆盖前端 **Phase 1**，目标是让用户可以在同一页面完成：
- 输入本地 PDF 路径并导入论文
- 手动触发解析
- 手动触发摘要生成
- 查看当前状态、成功提示、错误提示
- 继续查看 AI 摘要与正文内容

## 2. 范围

### 2.1 本次要做
- 将当前深色 UI 改为浅色简洁工作台风格
- 在左侧栏顶部增加导入表单
- 保留论文列表，并补充选中态与更明确的信息层级
- 在右侧详情区增加操作条：解析、生成摘要、刷新
- 增加基础反馈能力：加载态、空态、成功提示、错误提示
- 操作完成后自动刷新论文列表与当前详情
- 保持当前后端 API 契约不变，继续使用“本地 PDF 路径文本输入”方式导入

### 2.2 本次明确不做
- 文件上传控件
- 拖拽上传
- 批量导入
- 自动串行处理（导入后自动解析/摘要）
- 搜索、筛选、排序
- 章节导航
- 正文高亮、锚点定位
- 任务轮询与实时进度条
- 多页面拆分
- 新状态管理库或新 UI 组件库

## 3. 设计原则

### 3.1 KISS
优先利用现有双栏布局与现有详情阅读区，只在必要位置补充操作和反馈，不重做页面架构。

### 3.2 YAGNI
只实现当前后端已经提供的最小动作：导入、解析、摘要、查看详情。所有上传体系、复杂任务调度、筛选检索都留到后续阶段。

### 3.3 DRY
列表刷新与详情刷新统一走 API 层，不做重复状态同步逻辑；动作成功后统一触发“刷新列表 + 刷新当前详情”。

### 3.4 单一职责
各组件职责清晰：导入表单负责输入与提交，列表负责选择，操作条负责动作触发，详情区负责阅读与摘要展示，反馈组件负责展示系统反馈。

## 4. 页面结构

### 4.1 总体布局
保留当前左右双栏布局：
- **左侧栏**：导入与论文选择
- **右侧主区**：当前论文详情、操作与阅读

### 4.2 左侧栏结构
从上到下依次为：
1. 页面标题
2. 导入表单
3. 论文列表

#### 导入表单字段
- `title`
- `source`
- `local_pdf_path`
- 提交按钮：`导入`

#### 论文列表展示信息
- 标题
- 来源
- 状态标签
- 选中态高亮

### 4.3 右侧主区结构
从上到下依次为：
1. 当前论文标题与基础信息
2. 操作条
3. 反馈区
4. AI 摘要卡片
5. 正文阅读区

#### 操作条按钮
- `解析`
- `生成摘要`
- `刷新`

### 4.4 视觉方向
- 页面背景改为白底或浅灰底
- 卡片为白色，使用轻边框与轻阴影
- 主操作按钮使用单一强调色
- 状态标签改为浅底配色
- 不做重装饰，优先工具感与可读性

## 5. 交互流程

### 5.1 导入流程
1. 用户填写标题、来源、本地 PDF 路径
2. 点击“导入”
3. 导入中按钮进入禁用态
4. 导入成功后：
   - 清空错误提示
   - 展示成功提示
   - 刷新列表
   - 默认选中新导入论文
   - 拉取该论文详情并展示在右侧
5. 导入失败后：
   - 保留用户输入
   - 展示错误提示
   - 不清空当前右侧详情

### 5.2 解析流程
1. 用户选中某篇论文
2. 点击“解析”
3. 按钮进入加载态
4. 调用 `/papers/{id}/parse`
5. 成功后刷新列表与详情
6. 失败后展示错误提示

### 5.3 摘要流程
1. 用户选中某篇论文
2. 点击“生成摘要”
3. 按钮进入加载态
4. 调用 `/papers/{id}/summarize`
5. 成功后刷新列表与详情
6. 失败后展示错误提示

### 5.4 刷新流程
- 用户点击“刷新”后重新拉取当前详情
- 若当前未选中论文，则按钮不可用或不显示

## 6. 状态与反馈

### 6.1 前端状态
前端使用 React `useState` 管理，不引入额外状态库。

建议维护的状态：
- `papers`
- `selectedPaperId`
- `detail`
- `isLoadingList`
- `isSubmittingImport`
- `isRunningParse`
- `isRunningSummarize`
- `feedbackMessage`
- `errorMessage`

### 6.2 空态
- 列表为空时：左侧显示“还没有论文，请先导入”
- 未选中论文时：右侧显示“请选择左侧论文，或先导入新论文”

### 6.3 成功反馈
- 导入成功：提示“导入成功”
- 解析成功：提示“解析完成”
- 摘要成功：提示“摘要生成完成”

### 6.4 错误反馈
- 导入失败：显示后端错误文案或统一友好文案
- 解析失败：显示“解析失败，请稍后重试”
- 摘要失败：显示“摘要生成失败，请稍后重试”

### 6.5 加载态
- 表单提交时禁用导入按钮
- 解析中禁用解析按钮
- 摘要中禁用摘要按钮
- 列表首次加载时显示轻量占位或文本提示

## 7. 组件拆分

建议新增或调整以下组件：

### 7.1 `ImportForm`
职责：
- 管理导入表单输入
- 提交导入请求
- 向父组件回传导入结果

### 7.2 `PaperList`
职责：
- 展示论文列表
- 响应选中切换
- 高亮当前选中项

### 7.3 `PaperActions`
职责：
- 对当前选中论文触发解析、摘要与刷新操作
- 管理按钮禁用态和加载态

### 7.4 `FeedbackBanner`
职责：
- 展示成功消息与错误消息
- 可放在右侧主区顶部

### 7.5 `PaperDetail`
职责：
- 展示标题、摘要、正文
- 保持阅读区职责清晰，不直接承担请求逻辑

### 7.6 `SummaryCard`
职责：
- 展示摘要结构化字段
- 仅负责呈现，不负责状态控制

## 8. API 设计

前端 `lib/api.ts` 在现有查询能力上补齐以下动作：
- `fetchPapers()`
- `fetchPaperDetail(id)`
- `importPaper(payload)`
- `parsePaper(id)`
- `summarizePaper(id)`

### 8.1 刷新策略
所有写操作完成后，统一执行：
1. 刷新列表
2. 若当前有选中论文，则刷新当前详情

不做 optimistic update，不做轮询，不做本地推导状态。

## 9. 测试范围

### 9.1 交互测试
至少新增以下测试：
1. 导入成功后刷新列表并自动选中新论文
2. 点击“解析”后调用解析 API，并刷新详情
3. 点击“生成摘要”后调用摘要 API，并刷新详情与摘要内容
4. 导入失败时显示错误提示且保留输入内容
5. 未选中论文时右侧展示空态提示

### 9.2 回归要求
已有“点击列表项显示摘要与正文”的测试必须继续通过。

## 10. 实施顺序建议

建议按以下顺序落地：
1. 先改浅色布局与卡片样式
2. 增加导入表单与 API 方法
3. 打通导入后刷新列表与自动选中
4. 增加解析/摘要操作条
5. 增加反馈组件与错误处理
6. 补测试并回归验证

## 11. Phase 2 预留方向

本次不实现，但后续可顺延到 Phase 2：
- 搜索、筛选、排序
- 更强的阅读布局
- 章节导航
- 正文定位与高亮
- 更完整的任务状态呈现

## 12. 最终结论

本次前端 Phase 1 改版采用“**浅色可操作工作台**”方案：
- 不推翻当前 MVP 页面结构
- 以最小改动补齐导入、解析、摘要、反馈能力
- 把当前偏静态的阅读页升级为可操作的论文处理工作台
- 为后续 Phase 2 的搜索管理与阅读增强留出清晰扩展点
